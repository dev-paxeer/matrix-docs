## Overview

This section defines how changes are proposed, reviewed, versioned, and licensed in matrix-core. It combines workflow rules, review gates, changelog conventions, a community code of conduct, and the project's legal terms so contributors can move from idea to merged change with clear expectations.

The policy set is split between English source-of-truth files and Chinese-language mirrors under `rules/zh/`. The common workflow and code review documents define the baseline process, while the changelog, code of conduct, and license files govern release notes, community behavior, and distribution rights.

## Contribution Workflow

The workflow is defined in `rules/common/development-workflow.md` with Chinese mirrors in `rules/zh/development-workflow.md`.

### Required workflow sequence

1. **Research and reuse first.** Search for existing implementations before writing new code. The document requires GitHub code search first (`gh search repos`, `gh search code`), then library documentation (Context7 or primary vendor docs), then broader web research only when needed. Check package registries (npm, PyPI, crates.io) before writing utility code.

2. **Plan before coding.** Generate planning artifacts (PRD, architecture, system_design, tech_doc, task_list) before implementation. Break work into phases. Identify dependencies and risks.

3. **Use TDD.** Write tests first (RED), implement to pass tests (GREEN), refactor (IMPROVE), verify 80%+ coverage.

4. **Run code review immediately after coding.** Use the code-reviewer agent right after writing code. Fix CRITICAL and HIGH issues.

5. **Commit and push with disciplined history.** Follow Conventional Commits format. See `rules/common/git-workflow.md` for commit message format and PR process.

6. **Complete pre-review checks before requesting review.** All automated checks must pass, merge conflicts must be resolved, and the branch must be up to date with the target branch.

## Git Workflow

`rules/common/git-workflow.md` defines the commit message format:

```
<type>: <description>

<optional body>
```

Types: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `ci`.

For PRs: analyze full commit history, use `git diff [base-branch]...HEAD` to see all changes, draft a comprehensive PR summary with a test plan, and push with `-u` flag for new branches.

## Code Review Policy

`rules/common/code-review.md` and `rules/zh/code-review.md` define review expectations. Review is mandatory after code changes and before merges, with extra scrutiny for security-sensitive work.

| Area | Rule |
| --- | --- |
| When review is required | After writing or modifying code, before commits to shared branches, when security-sensitive code changes, when architecture changes, before merging PRs |
| Pre-review conditions | Automated checks must pass, merge conflicts resolved, branch up to date with target |
| Review checklist | Readable and well-named code, focused functions (<50 lines), cohesive files (<800 lines), shallow nesting (<4 levels), explicit error handling, no hardcoded secrets, no console logging or debug statements, tests for new functionality, 80%+ coverage |
| Security review triggers | Auth/authz code, user input handling, database queries, file system operations, external API calls, cryptographic operations, payment/financial code |
| Severity handling | CRITICAL blocks merge, HIGH should be fixed before merge, MEDIUM is a maintainability concern, LOW is optional |

### Review agents

| Agent | Purpose |
| --- | --- |
| `code-reviewer` | General code quality, patterns, best practices |
| `security-reviewer` | Security vulnerabilities, OWASP Top 10 |
| `typescript-reviewer` | TypeScript/JavaScript specific issues |
| `python-reviewer` | Python specific issues |
| `go-reviewer` | Go specific issues |
| `rust-reviewer` | Rust specific issues |

### Review workflow

1. Run `git diff` to understand changes.
2. Check security checklist first.
3. Review code quality checklist.
4. Run relevant tests.
5. Verify coverage >= 80%.
6. Use appropriate agent for detailed review.

## Coding Style

`rules/common/coding-style.md` defines the baseline style conventions:

- **Immutability**: Always create new objects, never mutate existing ones.
- **KISS**: Prefer the simplest solution that actually works.
- **DRY**: Extract repeated logic into shared functions.
- **YAGNI**: Do not build features or abstractions before they are needed.
- **File organization**: Many small files over few large files. 200-400 lines typical, 800 max. Organize by feature/domain, not by type.
- **Error handling**: Handle errors explicitly at every level. Never silently swallow errors.
- **Input validation**: Validate all user input before processing. Use schema-based validation where available.
- **Naming**: `camelCase` for variables/functions, `PascalCase` for interfaces/types/components, `UPPER_SNAKE_CASE` for constants, `use` prefix for hooks.

## Changelog Policy

`CHANGELOG.md` follows Keep a Changelog format with Semantic Versioning. Cross-references in parentheses point to `knowledge/matrix.kvx` session entries (e.g., `sess#NN`) or research decision IDs (`D1` through `D18`). The changelog records added features, removed items, versioned releases with dates, and implementation notes that point back to sessions and decisions.

## Code of Conduct

`CODE_OF_CONDUCT.md` adopts Contributor Covenant 2.1. Reports go to `conduct@paxeer.app`. Enforcement outcomes: Correction, Warning, Temporary Ban, Permanent Ban.

## Licensing

`LICENSE.md` is a custom Matrix-Protocol license with explicit copyleft, commercial-trigger, audit, and termination terms.

| Clause | Behavior |
| --- | --- |
| Grant | Unmodified source and object use, copy, and distribution under the license |
| Pure Caller Use | Integration-only use through published ABIs, APIs, or RPC endpoints without distributing modified work |
| Copyleft | Modifications and extensions must be released under the same license |
| Commercial Trigger | Required if charged fees exceed USD 100,000 in any rolling year or liquidity under control exceeds USD 10,000,000 |
| Contact duty | Crossing a trigger requires contacting `license@Paxeer.app` within 15 days |
| Audit | Paxlabs may request a yearly audit and an additional for-cause attestation |
| Termination | Material breach not cured within 15 days terminates the license |
| Governing law | New York law, venue in New York County |
